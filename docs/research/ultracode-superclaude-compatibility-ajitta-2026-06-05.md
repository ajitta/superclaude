# Compatibility and Integrated Use of ultracode (Workflow harness) with the SuperClaude content framework
## 한줄 요약 (Korean TL;DR)

- **호환된다. 서로 경쟁하지 않는다.** ultracode/Workflow는 Claude Code의 **하니스(harness) 실행 계층** 기능(`/effort ultracode`, 세션 한정)이고, SuperClaude는 `~/.claude/`에 설치되는 **콘텐츠 프레임워크**(마크다운 + 파이썬 훅)다. 서로 다른 계층에서 작동하므로 **직교(orthogonal)** 관계 — 합성(compose)되지 합쳐서 충돌하지 않는다.

## 1. Framing — Two Layers, Not Two Competitors

SuperClaude and ultracode answer different questions and live at different layers of the stack. Conflating them is the central error this report guards against.

**SuperClaude is a CONTENT framework.** Per the project `CLAUDE.md` (Architecture) and `src/superclaude/ARCHITECTURE.md:7-17`, it is "markdown files ... installed into `~/.claude/` to configure Claude Code's behavior." It ships content through three mechanisms (`ARCHITECTURE.md:117-123`): (1) always-loaded core via the `CLAUDE_SC.md` @import chain (`CLAUDE_SC.md:1-3` = `@core/FLAGS.md / @core/PRINCIPLES.md / @core/RULES.md`); (2) on-demand modes/MCP injected by `context_loader.py` (UserPromptSubmit hook, 8K budget, `context_loader.py:46-55`); (3) CC-native agents/commands/skills routed by the harness. **SuperClaude owns no scheduler, no process supervisor, no parallel executor.** Its `--delegate` auto-thresholds are documentation-only — `context_loader.py:501-529` `_EXECUTION_DIRECTIVES` has keys *only* for `--iterations`, `--loop`, `--concurrency`, `--plan`; there is **no `--delegate` key**, so `--delegate` emits no runtime behavioral directive at all (verified; memory obs 4982/5282).

**ultracode/Workflow is a HARNESS feature.** Per the authoritative task description, `/effort ultracode` is a session-only Claude Code effort level (xhigh thinking + standing opt-in to author/run dynamic Workflows). The `Workflow` tool runs a deterministic JS orchestration script in the background over MANY subagents (`pipeline()`, `parallel()`, `agent()`, `runId`, resume-from-cache). **None of these primitives appear anywhere in SuperClaude content** — a repo-wide grep for `Workflow(|runId|pipeline(|agentType|resumeFromRunId` matches only the unrelated `EnterWorktree` token in `RULES.md`. SuperClaude's own roadmap research treats "Dynamic workflows" and "Effort control" as explicitly **out of scope / monitor-only** (`docs/features/opus-4-8-alignment/02-research.md:83,104`), independently confirming the layer boundary.

**Why this makes them orthogonal.** The fan-out spawn boundary is the dividing line. SuperClaude decides *whether and with what intent* to delegate (policy, `RULES.md <sub_agent_decision>`); Workflow performs the *actual process-level fan-out* (mechanism). They do not contend for the same control point — a Workflow `parallel()`/`pipeline()` fans out without any SuperClaude `--delegate` flag ever being set (verified OV-1, confirmed verdict). The relationship is **compose, not compete**.

## 3. Conflicts & How to Reconcile

The Analyze phase (memory obs 5269) found **no irreducible conflict** — every apparent conflict reconciles through the layer distinction plus SC's priority chain `Safety > Scope > Restraint > Quality > Speed` (`RULES.md:8`).

**3.1 "Run a Workflow for every substantive task" vs `<sub_agent_decision>` "Never sub-agent" (`RULES.md:15`).** *Not a contradiction* — but **not** because the two share an exemption set (C1 nuanced correction). They govern different primitives at different layers: `<sub_agent_decision>` decides whether to delegate ONE subagent via the Agent tool; ultracode decides whether to author a multi-subagent Workflow. Their carve-outs actually *differ*: ultracode exempts only "conversational/trivial mechanical turns" (matching just the "<30s direct" trigger), but **embraces** "sequential A→B" work as sequenced workflows ("multi-phase work = several workflows in SEQUENCE") — the exact case `<sub_agent_decision>` lists under "Never sub-agent." Reconciliation: they are orthogonal on the axis of *which primitive*, not on coinciding exemptions.

**3.2 ultracode "be maximally exhaustive" vs R06 Scope / R18 Necessity Test (`RULES.md:41,48`).** A real but **reducible** tension, not irreducible (C4 nuanced correction). ultracode's directive textually targets the ANSWER-QUALITY axis ("exhaustive/correct answer, not fastest/cheapest" — a *cost* contrast), and SuperClaude's own content uses "exhaustive" only in the depth sense (deep-researcher 5+ hops, RESEARCH_CONFIG 50+/5/5). It is reconciled by **two** mechanisms: (1) the verified priority chain `RULES.md:8` with R18 at 🔴-tier deterministically subordinates any breadth-expansion reading; (2) subprocess FS writes are discarded, so a workflow **cannot create unsolicited files itself** — it returns markdown and the MAIN LOOP (where R06/R18 are always-loaded via @import) performs every Write, structurally routing deliverable decisions through R06/R18-governed code. Residual: the harness prose does not *textually* pin "exhaustive" to depth-only, so the depth-not-breadth reading is enforced by SC content + file-write routing, not harness prose — a **soft governance gap**, not a hard conflict.

**3.3 Token posture: ultracode "cost not a constraint" vs `--uc`/`--token-efficient` (`FLAGS.md:13,43`).** Orthogonal posture difference, not a same-axis collision (C2 nuanced correction). They optimize different variables (PROCESS breadth vs OUTPUT compression under CONTEXT pressure). Resolution: exhaustive process + `--uc`-compressed transport. The SC economy **safety floor survives** ultracode at the ≥85% zone because ultracode never claims context-window authority and `Safety > optimization` (`FLAGS.md:49`). **Caveat (C3 nuanced correction):** the 85% trigger is NOT a "hard floor" — there is no runtime code reading `context_window.used_percentage`; `context_loader.py:114` only injects `MODE_Token_Efficiency.md` on the literal flag string. It is **model-followed prose**, same enforcement class as `--delegate` thresholds. And in the ≥60% *proactive* band the postures genuinely **invert** — so the relationship is "orthogonal at the safety floor, posture-inverted in the proactive band," not flat "no conflict" everywhere.

**3.4 Default-inversion for analysis sessions: `FLAGS.md:39` vs ultracode default.** `FLAGS.md:39` ("analysis/discussion sessions rarely need `--delegate`/`--loop`; use `--seq`/`--tavily`/`--c7` instead") biases analysis turns toward solo; ultracode defaults to fanning out. Reconcilable by precedence (C5 nuanced correction): a user-chosen ultracode session legitimately governs because the user explicitly opted into a harness effort level — *in spirit* consistent with `FLAGS.md:50` "Explicit Override: user flags > auto-detection," though that rule is intra-content-layer and applies here only by analogy. Note: even under ultracode the Workflow's subagents still reach `--seq`/`--tavily`/`--c7` (via ToolSearch), so the only thing reversed is solo-vs-multi-agent *mechanism*, not tool selection.

## 4. Overlaps & Which to Prefer

**4.1 Iteration loops (the one genuinely duplicative intent).** SC `--loop`/`--iterations` and Workflow `loop-until-dry`/`budget{}` both encode "iterate until no improvement" (OV-5 nuanced). **Prefer the harness loop for execution** under ultracode (deterministic termination + token-budget object); **keep SC's R20 verifiable success criteria** as the quality gate the loop terminates against. No documented precedence rule exists between them — this is reasonable inference, not stated fact. (Note OV-5's correction: `--loop`/`--iterations` is *a* duplicative pairing, not provably *the only* one.)

**4.2 Output-shape enforcement: `opts.schema` vs Delegate-packet OUT trust.** Complementary layers, but they STACK on the return direction (not "opposite directions") — OV-3 nuanced. `opts.schema`/StructuredOutput enforces the **SHAPE/TYPE** of the returned object; it **cannot mechanically enforce `required_evidence_format`** in the sense SC means (real, re-validatable file:line). A schema validator cannot detect a fabricated-but-well-typed citation — which is exactly why the context-leak gotcha mandates re-grep before acting (memory obs 5283). **Prefer:** use `opts.schema` to harden return shape AND keep SC's prose re-validation for evidence *truth*. The harness supersedes SC only on shape, never on evidential trust.

**4.3 Verification structure: adversarial-verify/judge-panel vs `verification_ladder` + self-review.** Different granularities, no collision (OV-4 confirmed). SC's self-review is ONE read-only advisory agent (`self-review.md:6` tools = Read/Grep/Glob/Agent, no Write) with recall-first finding policy — NOT an N-skeptic kill-on-majority mechanism. **Prefer:** harness for the parallel adversarial *structure*; SC for per-finding policy (severity/confidence tagging) + blast-radius gate deciding *when* verification is required. `/sc:spec-panel` + `--vs` are the nearest SC analogs but are single-session output-diversity, not multi-process skeptics.

**4.4 `scripts/parallel_ab` vs Workflow.** Not duplicative — different process levels, never contend (OV-6 confirmed). `parallel_ab` is an **offline** A/B-eval harness (`orchestrator.py:86-90` asyncio.gather over `claude -p` subprocesses, cap `AB_MAX_PARALLEL=8` hard-reject, emits matrix.md+decision.md); it calls **no** harness Workflow/Task/EnterWorktree tool. Workflow orchestrates agents *inside* one session. Use `parallel_ab` for prompt/model A/B comparison; use Workflow for runtime delegation.

## 5. Complements & Synergies (SC as governance layer riding inside Workflow)

**5.1 SC fills the harness's intent-fidelity gap.** The harness `agent(prompt, opts?)` takes a free-form prompt with no built-in intent guarantee. SC's 6-field Delegate packet (`RULES.md:18`) supplies the missing prompt-content contract. **Precise framing (SYN-1 nuanced):** this is six content fields packed into ONE free-form prompt argument (many-to-one), not a one-to-one field↔parameter mapping; `allowed_scope`/`forbidden_changes` have no `opts` counterpart and remain prose discipline.

**5.2 Verbatim intent propagation is a governance NORM the harness lacks.** `RULES.md:9` + `:132` mandate the subagent receive the user request verbatim, no re-interpretation. The harness does NOT auto-inject the user turn — `agent-authoring.md:13` confirms a subagent receives "only markdown body ... not parent system prompt" — so an orchestrator must explicitly embed the verbatim request. **Caveat (SYN-2 nuanced):** this is a *norm/contract* bridge, not a claim the engine is mechanically incapable of forwarding intent (a script could use `initialPrompt`); SC's distinctive contribution is the discipline, not the only delivery mechanism.

**5.3 Trust-boundary governance makes fast fan-out safe.** `RULES.md:18` + context-leak gotcha make every `agent()` return **advisory-only**, requiring the main loop to re-grep/re-read cited file:line before acting. This counters the engine's speed incentive. **Caveat (SYN-3 nuanced):** the engine is not validation-free (`opts.schema` shape-check, scriptable adversarial-verify) and Opus 4.8 has native self-checking (`02-research.md:34,90`: ~4x fewer unremarked flaws), so summaries wouldn't be trusted "blindly" absent SC. SC's distinct contribution is making grounding-revalidation **mandatory and uniform** rather than optional/scripted.

**5.4 The clearest synergy — SC's documented enforcement gap is the harness's reason to exist.** SC ships no in-session delegation scheduler (verified: `context_loader.py:501-529` no `--delegate` directive; `<sub_agent_decision>` is model-followed heuristic). The Workflow tool is the deterministic executor SC explicitly lacks, while SC's `<sub_agent_decision>` still governs *whether* to author a workflow at all (SYN-6 nuanced — note the cited "doc-only" finding spans two different threshold sets: FLAGS numeric vs RULES qualitative).

**5.5 Effort uniformity for free.** ultracode session-level xhigh lifts every SC agent because no SC agent hardcodes `effort:` (verified: 0 grep matches in `src/superclaude/agents/`; commit `8edd05d` stripped it from the 11 that had it). Precedence is `env > frontmatter > session > model default` — session xhigh propagates uniformly. This *validates* SC's inherit-only stance (IR-5 nuanced). **Caveat:** uniformity is a current-state property, not a structural invariant — frontmatter outranks session, so a future per-agent `effort:` override would silently break it.

## 8. Bottom-Line Recommendation

**ultracode and SuperClaude are compatible and complementary — use them together, with ultracode as SuperClaude's missing deterministic execution engine and SuperClaude as the governance layer riding in the main loop.**

1. **Adopt the layer split deliberately.** Let SuperClaude content decide *whether and with what intent* to fan out (`<sub_agent_decision>`, Delegate packet, verification_ladder); let the Workflow tool perform the *fan-out execution* it provides and SC lacks. This is the single highest-value, lowest-risk pairing (SYN-6).

2. **Keep all writes and approvals in the main loop.** Document-producing workflows RETURN markdown; the main loop writes per `doc_output_convention` (default `docs/features/<slug>/`) and runs the `>3-file` approval checkpoint (subagents cannot pause). This satisfies the file-write friction and R06/R18 simultaneously.

3. **Harden returns mechanically, but never trust evidence by shape.** Use `opts.schema` for return-shape, and ALWAYS keep SC's mandatory grounding-revalidation (re-grep cited file:line) for evidence *truth* — `opts.schema` cannot detect a well-typed fabricated citation (memory obs 5283).

4. **The safety story is now settled for hooks (verified 2026-06-05, Appendix A).** SC's enforced PreToolUse/PostToolUse hooks (`file_size_guard`, destructive-Bash blocker, `loop_guard`) FIRE in every subagent type including worktree-isolated — so the safety bridge holds even in isolated fan-out, and independent of whether the subagent's prompt carries RULES.md (hooks act at the tool-call boundary). Two residual caveats remain: (i) `context_loader` mode/MCP injection is main-loop-only — if a subagent needs `--seq`/`--c7`/`--tavily` or mode context, pass it explicitly in the `agent()` prompt (it is NOT auto-injected); (ii) the `loop_guard` state file is lock-free — concurrent same-cwd subagents can lose breaker counters (fail-open, no crash; worktree subagents are path-isolated). Still gate resume-from-cache through R15 (7.5).

5. **Set expectations on caps and tokens.** `--concurrency` does not buy process parallelism; the `min(16, cpu-2)` harness cap governs. Under ultracode keep the ≥60% proactive `--uc` as a context-window guard (it is not a cost concern), and run exhaustive *process* with compressed *transport*.
