## 1. Method & Scope

**What was audited.** The SuperClaude content framework at `C:\Users\ajitta\Repos\ajitta\superclaude` (current shipped version `4.6.0+ajitta` per `pyproject.toml:7` / `__init__.py:7`): the always-loaded core constitution (`core/FLAGS.md` + `PRINCIPLES.md` + `RULES.md`), the 7-hook enforcement layer (`hooks/hooks.json` + `scripts/*.py`), the install pipeline (`cli/install_components.py`, `install_paths.py`), 35 commands / 23 agents / 7 behavioral modes / 5 skills, the project's own self-critique docs (`docs/analysis/`, `docs/research/`, `docs/features/opus-4-8-alignment/`), and the `gotchas/` trap registry. The comparison target is **obra/superpowers** at the marketplace-pinned SHA `6fd4507`, confirmed current as **v5.1.0** (GitHub API: ~217.8K stars, pushed 2026-06-03).

**Layer discipline.** Throughout, I separate the **HARNESS layer** (Claude Code: which events fire, subagent fan-out caps, the Bash permission prompt) from the **CONTENT layer** (SuperClaude markdown + its install-time hooks). Several findings are harness-boundary facts that *cannot* be fixed in markdown; those are marked explicitly so the maintainer does not chase a markdown fix for a Claude Code constraint.

**Evidence and the R18 gate.** Every problem was put through SuperClaude's own **R18 Necessity Test**: a *real* problem needs a concrete failure scenario, quantitative evidence, or user-facing impact — not "would be nicer/safer." All ten candidate problems were adversarially verified (recorded as observation 5361). **Six were downgraded to "nuanced"** (core facts hold, framing corrected) and **one (P10) was effectively refuted**. This report applies those verdicts: refuted claims are dropped or softened, nuance corrections are folded in, and inflated magnitudes are restated with measured numbers.

**Balance disclaimer.** SuperClaude is a deliberately self-critical framework; most of its weaknesses are already documented in its own repo with file:line anchors and STATUS tags. I do not re-present solved issues as new findings, I credit what SuperClaude does genuinely well (Section 7), and I do not strawman superpowers — where the two simply chose different goals, I say so honestly rather than declaring a loser.

**One caveat on superpowers freshness.** superpowers is **not installed** as a live plugin on this machine — only SHA-pinned in the marketplace registry, with SuperClaude-adapted ports and a few standalone obra skills present locally. v5.1.0 facts come from the live GitHub API + RELEASE-NOTES (internally consistent across `plugin.json`, the contents API, and the changelog). ### 2.3 Missing problem CLASSES (this audit's net-new contribution)

The verified problems above are mostly *recalibrations* of things SuperClaude already half-knows. The genuinely under-covered findings, surfaced by the completeness critique, are three **problem classes** the existing self-critique never frames:

- **MP-SEC — Hook security posture is unaudited.** SuperClaude installs 7 hooks that run on **untrusted model output**, and `hooks.json:93` pipes model-controlled text (`.tool_input.command`) into a shell `grep -qE`. `superclaude install` writes executable hook commands into the user's `settings.json` (an arbitrary-code-on-every-tool-call surface), and `insight_writer.py` harvests model text into a persisted JSONL. No map asks whether any hook is injection-exploitable or what the trust boundary of that harvesting is. A framework whose headline safety claim is "PreToolUse blocks `rm -rf`" owes a security pass on **its own hook surface**. *R18: verified — model-controlled input demonstrably reaches a shell context (`hooks.json:93`); `install_settings.py` only merges/dedups, it never hardens the grep.* **Severity: medium, OPEN, net-new class.**
- **MP-ONBOARD — First-run UX compounds 4 silent-failure points.** A stock-Windows new user following the README hits, in one install: (a) the silently-inert destructive guard (P1), (b) possible `make sync` failure (`windows-make-sync-broken`, os error 5), (c) a **Serena prerequisite the installer does not install** (`README:107` "install it yourself first"), and (d) `{{SCRIPTS_PATH}}` command-body templates that **ship unresolved** in installed commands (`insight.md` ×10). None is surfaced to the user, and `doctor` checks none of P1's deps. The audit's individual low-severity items hide an aggregate **first-run cliff**. **Severity: medium (as a class), OPEN.**
- **MP-TESTCOV — The test suite is structure-heavy, safety-behavior-light.** The "~1,904-test suite" is repeatedly credited as a maturity counterweight, but coverage is dominated by `test_*_structure.py` shape-lint. Of the 3 enforced safety hooks, 2 have tests (`test_file_size_guard.py`, `test_loop_guard.py`) and the **one with no Python implementation also has no test** — `test_hooks.py` has zero destructive-command assertions (verified). There is also **no behavioral (`claude -p`) trigger regression in CI** (the keystone "when-to-use" bug shipped invisibly for ~14 days for exactly this reason; a canary fixture was built but left manual-run). Test coverage is concentrated **away from** the load-bearing safety surface. **Severity: medium, OPEN.**

### Fair narrative

The two frameworks are **not competitors on the same axis** — they are opposite bets that happened to converge on the same engineering spine (brainstorm → spec → plan → execute → verify), which is itself mutual validation that the methodology is sound. SuperClaude bets on **deterministic, always-present capability breadth**: 35 commands and 23 domain experts under a numbered constitution, with deep Claude-Code integration (real PreToolUse safety hooks a portable skill set cannot assume). superpowers bets on **process consistency at near-zero standing cost**: a tiny SessionStart gate that makes the right workflow fire by default, portable across 8 harnesses, in the official marketplace.

Where superpowers clearly wins — enforcement model, token economy, marketplace reach, maintenance-surface coherence — it wins **because of its narrowness**, the very thing that makes it weaker for multi-domain delegation. Where SuperClaude wins — domain breadth, CC-native safety, pytest-plugin + structural-lint maturity — it wins **because of its breadth**, the very thing that creates its standing-tax and drift costs. The honest verdict on most dimensions is "different-goals," with three genuine superpowers leads (enforcement gate, token economy, distribution) and one genuine tie (governance: SC thinks richer, SP executes more reliably).

**Do not idealize superpowers.** The single strongest evidence in this whole comparison — SP's measured removal of its review-loop — is double-edged: SP shipped that ~25-minute ceremony for **five versions** before measuring and cutting it. So SuperClaude's lack of delegation measurement is a gap superpowers **also had until recently**, not a discipline superpowers has always exemplified. The lesson is the *method* (measure ROI, discriminate execution from review delegation), not a halo.

## 6. What SuperClaude should KEEP (real strengths — do not throw out)

- **The always-loaded numbered constitution** is a genuine, deterministic behavioral prior — its weakness (standing tax) is a *deliberate, documented* trade-off, not a defect. The on-demand tier (3-tier injection, 8K budget, dedup) is real token-discipline engineering.
- **The enforced hooks that exist propagate correctly.** Empirically verified (research Appendix A, 4/4): `file_size_guard` and `loop_guard` fire in every subagent type including worktree-isolated, *independent* of whether the subagent's prompt carries RULES.md, because hooks act at the harness tool-call boundary. The safety **floor** is real even in isolated fan-out — the architecture's strongest property.
- **Domain-expert agent breadth** (23 specialists, tight 75-103-line bodies) genuinely beats a single general-purpose Task for multi-domain delegation — SuperClaude's clearest advantage over superpowers.
- **Exceptional self-awareness:** the `gotchas/` trap registry (it documents its *own* hallucination/over-engineering/Windows-sync/compaction-drift hazards), honest self-critique docs with file:line + severity + STATUS, and disciplined restraint (Opus-4-8 alignment **refused** to weaken safety practices on vendor claims per R18).

This is the cheap machinery that makes the I-series fixes low-risk.

(The destructive-Bash hook's fail-open is the one place this posture is a liability, not a virtue — hence I1.)

## 7. Risks & Open Questions

- **Inferred vs documented.** P2 ("context_loader not firing per-subagent") is **inferred-strong** in the research doc (Appendix A.3), not empirically isolated the way the 4/4 hook-firing test was — treat as high-confidence inference, not measured fact. MP-SEC (injection-exploitability of `hooks.json:93`) is an **un-tested hypothesis**: I verified model-controlled text reaches a shell `grep`, but I did **not** construct or run an exploit — the security pass (I10) is needed to confirm or dismiss it.
- **Harness-boundary, not markdown-fixable.** P2, P7's fan-out cap, and the P6 race are **Claude Code** facts (subagent events, `min(16,cpu-2)`, lock-free concurrency). The maintainer should not chase a markdown fix for these; the correct responses are documentation (I8), honest flag wording (I7), and an atomic-write code fix (I2) respectively. A separate latent risk: harness `resumeFromRunId` replays **cached** agent results that can cite stale file:line — the exact failure R15/R02 prevent, reintroduced by a harness optimization; gate resume through R15 revalidation.
- **Stale-superpowers uncertainty.** All v5.1.0 facts come from the live API + RELEASE-NOTES (internally consistent), but superpowers is **not** installed locally — the comparison rests on the SHA-pinned upstream + adapted ports, not a running install. Skill **counts** (14) are firm; the older "brainstorm WebSocket server" detail was **not** re-confirmed — do not rely on it. SP moves fast (217.8K stars, pushed 2026-06-03); re-verify before any decision that depends on a precise v5.1.0 surface detail.

- **Completeness caveat (this audit).** This report is a synthesis of pre-computed MAPs + adversarial verdicts plus targeted re-verification of the highest-leverage facts (`hooks.json:93`, `context_loader.py:501-529`, the skill-count, doctor/test gaps, `ship/SKILL.md:10`). I did **not** independently re-run the full ~1,904-test suite, dynamically exploit any hook, or measure the delegation ROI that I9 recommends — those remain open work, not findings.

## 8. Bottom Line

**SuperClaude is a healthy, unusually self-aware content framework whose real problems are mostly already known and mostly low-severity — but it has exactly one safety defect worth fixing now, three under-framed problem CLASSES, and one strategic lesson to borrow.**

- **Fix now (cheap, high-value):** the P1 destructive-guard Python rewrite + test (I1) — a *safety* hook that silently no-ops on its own primary test platform is the single most actionable item. Pair it with the loop_guard atomic write (I2), the dangling-ref inlining (I3), and a version SSOT + drift lint (I6). All four are low-risk on existing test infra.
- **Investigate (net-new classes):** the hook **security** posture (MP-SEC/I10), the compounding **onboarding** cliff (MP-ONBOARD/I11), and the **safety-behavior test coverage** gap (MP-TESTCOV/I12). These are this audit's genuine contribution beyond restating SuperClaude's own self-critique.
- **Borrow the method, not the halo (P9/I9):** measure delegation ROI the way superpowers did, and discriminate high-value execution delegation from low-value review delegation. superpowers earned this lesson the hard way (five versions of unmeasured ceremony) — take the method, not an idealized image.
- **Keep the thesis:** the always-loaded constitution, the domain-agent breadth, the CC-native enforced hooks, and the self-aware gotchas registry are SuperClaude's real differentiators. The enforced safety floor genuinely holds even in isolated fan-out. Do not trade breadth and integration depth for superpowers' narrowness — they are different goals, and SuperClaude's goal is a legitimate one. The right move is to make the prose/enforced boundary **honest** (I4), demote the demotable (I5), and harden the one safety hook that betrays the rest (I1).
